# DMN Implementation Design

## Overview

This document outlines the design and implementation plan for adding full DMN (Decision Model and Notation) capability to the TaktX engine. The implementation follows the exact same architectural patterns established by the BPMN processing support: XSD-driven JAXB generation, internal DTO mapping, and FEEL expression evaluation.

---

## Architecture Summary of the Existing BPMN Stack

Understanding the existing stack is the key to designing a consistent DMN counterpart.

| Layer | BPMN component | Role |
|---|---|---|
| XML Schema | `taktx-shared/src/main/resources/bpmn/*.xsd` | Defines the XML grammar (BPMN 2.0 + Zeebe extension) |
| JAXB classes | `io.taktx.bpmn.*` (generated by XJC) | Strongly-typed Java representation of the raw XML |
| Mapper | `io.taktx.xml.BpmnParser` + `BpmnMapper` hierarchy | Converts JAXB objects → internal DTOs |
| DTOs | `io.taktx.dto.*` | Internal, serializable representation used across the engine |
| Topics / Stores | `Topics`, `Stores` enums | Kafka topics and state stores |
| Definition processing | `DefinitionsProcessor`, `DefinitionsCache` | Deploys and caches process definitions |
| Instance processing | `ProcessInstanceProcessor` + `processor.*` package | Executes active instances |
| FEEL engine | `FeelExpressionHandler` / `FeelExpressionHandlerImpl` | Evaluates FEEL expressions |
| REST API | `ProcessDefinitionResource` | Exposes definitions over HTTP |

The DMN implementation mirrors every layer of this stack.

---

## DMN Concepts to Implement

| DMN Concept | Description |
|---|---|
| `Definitions` | Root element of a `.dmn` file, contains one or more `Decision` elements |
| `Decision` | Named decision unit; contains either a `DecisionTable` or a `LiteralExpression` |
| `DecisionTable` | Table of input columns, output columns and rules; governed by a hit policy |
| `InputClause` | Defines a single input column and its FEEL `inputExpression` |
| `OutputClause` | Defines a single output column (name + type) |
| `Rule` | One row in the table; pairs input entries with output entries |
| `InputEntry` / `OutputEntry` | Cell values; FEEL unary tests (input) or FEEL expressions (output) |
| `LiteralExpression` | Single FEEL expression producing a scalar output |
| `DecisionRequirementsDiagram` | (optional, later) Dependency graph between decisions |
| `Hit Policy` | UNIQUE, FIRST, ANY, COLLECT (+/min/max/count), RULE ORDER, OUTPUT ORDER, PRIORITY |

## Phase 1 – XSD Files and JAXB Generation

**Location:** `taktx-shared/src/main/resources/dmn/`

Add the following XSD files:

| File | Purpose |
|---|---|
| `DMN13.xsd` | Standard DMN 1.3 schema (https://www.omg.org/spec/DMN/20191111/MODEL/) |
| `DMNDI13.xsd` | DMN diagram interchange schema (used by tooling; parsing-only) |
| `DC.xsd` | Re-use or copy the diagram common schema (already present for BPMN) |

Configure XJC in `taktx-shared/build.gradle.kts` to generate a **second** set of JAXB classes from the DMN XSD into the package `io.taktx.dmn`.

Key generated classes:

```
io.taktx.dmn.TDefinitions         (root element)
io.taktx.dmn.TDecision
io.taktx.dmn.TDecisionTable
io.taktx.dmn.TInputClause
io.taktx.dmn.TOutputClause
io.taktx.dmn.TRule
io.taktx.dmn.TUnaryTests          (input entry)
io.taktx.dmn.TLiteralExpression   (output entry / literal decision)
io.taktx.dmn.THitPolicy           (enum)
io.taktx.dmn.TItemDefinition
```

No additional Zeebe-specific XSD is required for DMN – Zeebe/Camunda 8 uses standard DMN 1.3 without extension elements at the decision-table level. The only Zeebe-specific element is in BPMN (`zeebe:calledDecision` on `businessRuleTask`), which is already covered by the existing `zeebe.xsd`.

## Phase 2 – Internal DTOs

**Location:** `taktx-shared/src/main/java/io/taktx/dto/`

New DTOs follow the same Lombok + `@RegisterForReflection` conventions as existing DTOs.

### Decision model DTOs

```
DmnDefinitionsKeyDTO         id + hash (mirrors DefinitionsKey)
ParsedDmnDefinitionsDTO      result of parsing a .dmn file (mirrors ParsedDefinitionsDTO)
DmnDecisionDTO               id, name, key, type (TABLE or LITERAL)
DmnDecisionTableDTO          hitPolicy, inputs, outputs, rules
DmnInputClauseDTO            id, label, inputExpression (FEEL string), typeRef
DmnOutputClauseDTO           id, label, name, typeRef, defaultOutput
DmnRuleDTO                   id, inputEntries (List<String>), outputEntries (List<String>)
DmnHitPolicy                 enum: UNIQUE, FIRST, ANY, COLLECT, RULE_ORDER, OUTPUT_ORDER, PRIORITY
DmnCollectOperator           enum: NONE, SUM, MIN, MAX, COUNT
DmnLiteralExpressionDTO      FEEL expression string
```

### Definition registration DTOs (mirrors existing BPMN deployment DTOs)

```
DmnDefinitionsTriggerDTO     base sealed type (mirrors DefinitionsTriggerDTO)
XmlDmnDefinitionsDTO         wraps raw XML string (mirrors XmlDefinitionsDTO)
```

### BusinessRuleTask DTO (new flow element in BPMN)

```
BusinessRuleTaskDTO          extends ExternalTaskDTO
                             fields: decisionId, resultVariable
```

## Phase 3 – DMN Parser and Mapper

**Location:** `taktx-shared/src/main/java/io/taktx/xml/`

### `DmnParser` (mirrors `BpmnParser`)

Static utility:
1. Detects namespaces with SAX (for future extension compatibility)
2. Unmarshals XML into `io.taktx.dmn.TDefinitions` via JAXB
3. Computes SHA-256 hash of the raw XML
4. Delegates to `DmnMapper` to produce `ParsedDmnDefinitionsDTO`

### Mapper hierarchy

| Interface | Implementation | Purpose |
|---|---|---|
| `DmnMapper` | `GenericDmnMapper` | Top-level: iterates `TDecision` elements |
| `DecisionMapper` | `GenericDecisionMapper` | Maps `TDecision` → `DmnDecisionDTO` |
| `DecisionTableMapper` | `GenericDecisionTableMapper` | Maps `TDecisionTable` → `DmnDecisionTableDTO` |
| `LiteralExpressionMapper` | `GenericLiteralExpressionMapper` | Maps `TLiteralExpression` → `DmnLiteralExpressionDTO` |

### `DmnMapperFactory` (mirrors `BpmnMapperFactory`)

Responsible for instantiating the correct mapper implementations. Currently only `Generic*` variants exist; future Camunda-specific extensions (if any arise) can be added here by namespace detection.

### Zeebe `BusinessRuleTaskMapper`

| Interface | Implementation | Purpose |
|---|---|---|
| `BusinessRuleTaskMapper` | `GenericBusinessRuleTaskMapper` | Bare BPMN businessRuleTask (id + name only) |
|  | `ZeebeBusinessRuleTaskMapper` | Reads `zeebe:calledDecision` → `decisionId`, `resultVariable` |

`ZeebeBusinessRuleTaskMapper` is selected by `BpmnMapperFactory` when the Zeebe namespace is present (same pattern as `ZeebeServiceTaskMapper`).

`GenericFlowElementMapper.mapActivity()` gains a `TBusinessRuleTask` case that delegates to the mapper returned by `bpmnMapperFactory.createBusinessRuleTaskMapper()`.

## Phase 4 – Kafka Topics and State Stores

### Topics (added to `io.taktx.Topics`)

| Topic enum value | Topic name | Cleanup |
|---|---|---|
| `DMN_DEFINITIONS_TRIGGER_TOPIC` | `dmn-definitions` | `DELETE` |
| `XML_BY_DMN_DEFINITION_ID` | `xml-by-dmn-definition-id` | `COMPACT` |

### Stores (added to `io.taktx.engine.pd.Stores`)

| Enum value | Store name | Value type |
|---|---|---|
| `GLOBAL_DMN_DEFINITION` | `global-dmn-definition` | `DmnDefinitionDTO` (versioned parsed definition) |
| `DMN_VERSION_BY_HASH` | `dmn-version-by-hash` | `Map<String, Integer>` |
| `XML_BY_DMN_DEFINITION_ID` | `xml-by-dmn-definition-id` | `String` (raw XML) |

`DmnDefinitionDTO` is the stored form analogous to `ProcessDefinitionDTO`:

```
DmnDefinitionDTO
  DmnDefinitionsKeyDTO    key (id + hash)
  Map<String, DmnDecisionDTO>  decisions
  int                     version
  DmnDefinitionStateEnum  state (ACTIVE / INACTIVE)
```

`DmnDefinitionKey` mirrors `ProcessDefinitionKey` (id + version) and is used as the Kafka key for the definition store.

## Phase 5 – DMN Definition Processing

**Location:** `taktx-engine/src/main/java/io/taktx/engine/pd/`

### `DmnDefinitionsProcessor` (mirrors `DefinitionsProcessor`)

Kafka Streams `Processor<String, DmnDefinitionsTriggerDTO, Object, Object>`:
- Receives raw XML via `XmlDmnDefinitionsDTO`
- Calls `DmnParser.parse()` to get `ParsedDmnDefinitionsDTO`
- Computes a new version or finds an existing one (same hash-version-pair logic)
- Stores the versioned definition in `GLOBAL_DMN_DEFINITION`
- Forwards the raw XML to the `XML_BY_DMN_DEFINITION_ID` sink

### `DmnDefinitionsCache` (mirrors `DefinitionsCache`)

Lazy-loaded in-memory cache keyed by `(decisionId, version)`. The `BusinessRuleTaskInstanceProcessor` uses this cache to resolve a decision at execution time without a Kafka store lookup on the hot path.

## Phase 6 – DMN Execution Engine

**Location:** `taktx-engine/src/main/java/io/taktx/engine/dmn/`

### `DmnEvaluator` interface

```java
public interface DmnEvaluator {
    JsonNode evaluate(DmnDecisionDTO decision, VariableScope variables);
}
```

### `DmnEvaluatorImpl`

Uses the existing `FeelExpressionHandler` for all FEEL expression evaluation. Implements hit policy logic:

| Hit Policy | Behaviour |
|---|---|
| UNIQUE | Exactly one rule matches; returns single output object |
| FIRST | Returns output of the first matching rule |
| ANY | All matching rules must have identical output; returns that output |
| COLLECT (no operator) | Returns a list of all matching outputs |
| COLLECT + SUM/MIN/MAX/COUNT | Aggregates numeric outputs |
| RULE ORDER | Returns list in rule order |
| OUTPUT ORDER | Returns list sorted by output priority list |
| PRIORITY | Returns output with highest priority |

Evaluation loop (per rule):
1. For each input entry: evaluate the FEEL unary test against the variable in scope
2. If all input entries match: evaluate each output entry as a FEEL expression
3. Apply hit policy to collected results

FEEL unary tests and expressions are evaluated via `FeelExpressionHandlerImpl`. DMN unary test strings (e.g. `>= 100`, `"gold","silver"`) are evaluated by prepending `=` and wrapping in the FEEL engine's `evaluateUnaryTests` path if required, or by converting to boolean FEEL expressions.

### `DmnDecisionResolver`

Resolves a `decisionId` + optional `decisionVersion` to a `DmnDecisionDTO` using `DmnDefinitionsCache`. Throws a `DmnDecisionNotFoundException` if the decision is not found (triggers an incident in the process instance, analogous to an unresolvable expression).

## Phase 7 – BusinessRuleTask Integration in BPMN

### New model class

**`taktx-engine/src/main/java/io/taktx/engine/pd/model/BusinessRuleTask`** (mirrors `ServiceTask`):

```java
public class BusinessRuleTask extends ExternalTask {
    private String decisionId;
    private String resultVariable;
}
```

### New instance class

**`taktx-engine/src/main/java/io/taktx/engine/pi/model/BusinessRuleTaskInstance`** (mirrors `ServiceTaskInstance`).

### `BusinessRuleTaskInstanceProcessor`

**Location:** `taktx-engine/src/main/java/io/taktx/engine/pi/processor/`

Extends `ActivityInstanceProcessor`. On `processStartSpecificActivityInstance`:
1. Resolve the `DmnDecisionDTO` from `DmnDecisionResolver`
2. Invoke `DmnEvaluator.evaluate(decision, variableScope)`
3. Store the result in `variableScope.put(resultVariable, result)`
4. Set instance state to `COMPLETED` (fully synchronous – no Kafka round-trip, like `ScriptTask`)

On incident: if the decision is not found or evaluation fails, create an incident and leave the task in `FAILED` state.

### Registration

- Add `BusinessRuleTask` handling to `GenericFlowElementMapper.mapActivity()` (new `case TBusinessRuleTask` branch)
- Add `BusinessRuleTaskInstanceProcessor` to `FlowNodeInstanceProcessorProvider.getStateProcessorForActivity()`

## Open Questions / Design Decisions for Review

1. **DMN deployment channel**: Should DMN files be deployed through the same Kafka topic as BPMN (`definitions`) using a new `XmlDmnDefinitionsDTO` sub-type, or through a dedicated `dmn-definitions` topic? A dedicated topic keeps concerns cleanly separated and is recommended.

2. **Decision versioning**: DMN decisions are versioned independently of BPMN processes. A `businessRuleTask` should be able to specify a pinned `decisionVersion` (analogous to calling a specific version of a called process). The `zeebe:calledDecision` element already supports a `bindingType` attribute for this.

3. **Decision Requirements Diagrams (DRD)**: A DRD can chain decisions (the output of one feeds the input of another). Phase 1 implementation can treat each decision as self-contained. DRD chaining can be added in a follow-up once the basic table evaluation is working.

4. **Type coercion**: DMN `typeRef` values (`string`, `integer`, `long`, `double`, `boolean`, `date`, `time`, `date and time`, `duration`) should be coerced to/from FEEL types. This can reuse the FEEL engine's built-in type system.

5. **Standalone DMN evaluation API**: Should there be a REST endpoint to evaluate a decision directly (outside a BPMN process instance)? This would be useful for testing and ad-hoc queries. Recommended as a follow-up.

6. **Expression language negotiation**: DMN 1.3 allows other expression languages (JUEL, Groovy). TaktX will support FEEL only (same constraint as for BPMN script tasks), unless explicitly extended later.
